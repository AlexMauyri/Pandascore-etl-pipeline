# 🎮 Cybersport ETL Pipeline

Автоматизированный ETL-конвейер для сбора, обработки и визуализации данных о киберспортивных матчах из API [PandaScore](https://pandascore.co/). Данные ежедневно извлекаются, трансформируются и загружаются в аналитическое хранилище ClickHouse, а затем визуализируются через дашборды Superset.

## 🎯 О проекте

Проект решает задачу автоматизированного сбора данных о киберспортивных матчах, командах, турнирах, лигах и видеоиграх из внешнего API PandaScore с последующим построением аналитических дашбордов. Конвейер работает по расписанию (ежедневно), инкрементально подгружая только новые завершённые матчи, и сохраняет промежуточные результаты для обеспечения идемпотентности запусков.

Итоговые данные попадают в колоночную СУБД ClickHouse, оптимизированную под аналитические нагрузки, а визуализация метрик (количество матчей по дням, топ команд, распределение статусов, динамика турниров) реализована средствами Apache Superset.

## 🏗 Архитектура

Взаимодействие сервисов можно представить следующей схемой:

```
                         ┌────────────────────┐
                         │   PandaScore API    │
                         └──────────┬──────────┘
                                    │ HTTP (paginated)
                                    ▼
                         ┌─────────────────────┐
                         │  Airflow DAG        │
                         │  (extract →         │
                         │   transform → load  │
                         │   → cleanup)        │
                         └──────────┬──────────┘
                                    │
                 ┌──────────────────┼───────────────────┐
                 ▼                  ▼                   ▼
        ┌──────────────────┐ ┌─────────────────┐  ┌──────────────────┐
        │     MinIO        │ │   ClickHouse    │  │  PostgreSQL      │
        │ (raw/transformed │ │  (DWH: сущности │  │ (метаданные      │
        │   staging данные)│ │   + etl_run_log/│  │  Airflow)        │
        └──────────────────┘ │   etl_state)    │  └──────────────────┘
                             └───────┬─────────┘
                                     │
                                     ▼
                            ┌────────────────┐
                            │   Superset     │
                            │  (дашборды)    │
                            └────────────────┘

        ┌────────────────┐
        │     Redis      │  ← используется Superset для кэширования
        └────────────────┘
```

Компоненты Airflow (webserver/api-server, scheduler, triggerer, dag-processor) работают как отдельные контейнеры, разделяющие общую метабазу PostgreSQL и общий образ (`Dockerfile.airflow`, таргет `production`). MinIO выступает промежуточным буфером между этапами DAG — это позволяет перезапускать отдельные задачи без повторных обращений к внешнему API, если данные для соответствующего запуска (`dag_run_id`) уже сохранены.

## 🚀 Установка и запуск

### 1. Клонирование репозитория

```bash
git clone <URL_репозитория>
cd cybersport-etl-pipeline
```

### 2. Создание файла `.env`

Скопируйте пример конфигурации (если он есть в репозитории) или создайте `.env` в корне проекта вручную — см. раздел [Переменные окружения](#-переменные-окружения).

```bash
cp .env.example .env   # либо создайте файл вручную
```

### 3. Запуск сервисов

```bash
docker compose up -d --build
```

При первом запуске автоматически выполнится:
- инициализация схемы ClickHouse (`sql/init.sql`);
- инициализация метабазы Airflow (`sql/init-airflow.sql`);
- миграция базы Airflow через сервис `airflow-init`;
- инициализация Superset (создание админа, применение миграций).

### 4. Проверка состояния контейнеров

```bash
docker compose ps
```

Дождитесь статуса `healthy` для `clickhouse`, `postgres-airflow`, `minio` и `redis_rate_limiter` — остальные сервисы зависят от них и запустятся автоматически.

### 5. Активация DAG

Откройте веб-интерфейс Airflow (см. [Доступ к сервисам](#-доступ-к-сервисам)) и включите DAG `cybersport_etl_pipeline`.

## ⚙️ Переменные окружения

Ниже перечислены ключевые переменные, которые необходимо задать в `.env`:

**ClickHouse**
```
CLICKHOUSE_DB=etl
CLICKHOUSE_USER=etl_user
CLICKHOUSE_PASSWORD=<пароль>
```

**PostgreSQL (метаданные Airflow)**
```
AIRFLOW_POSTGRES_DB=airflow
AIRFLOW_POSTGRES_USER=airflow
AIRFLOW_POSTGRES_PASSWORD=<пароль>
```

**MinIO**
```
MINIO_ROOT_USER=<логин>
MINIO_ROOT_PASSWORD=<пароль>
MINIO_API_PORT=9000
MINIO_CONSOLE_PORT=9001
```

**Redis**
```
REDIS_PASSWORD=<пароль>
REDIS_URL=redis://:<пароль>@redis_rate_limiter:6379/0
```

**Superset**
```
SUPERSET__SQLALCHEMY_DATABASE_URI=<строка подключения к БД Superset>
SUPERSET_SECRET_KEY=<секретный ключ>
SUPERSET_ADMIN_USER=admin
SUPERSET_ADMIN_PASSWORD=<пароль>
SUPERSET_ADMIN_EMAIL=admin@example.com
```

**PandaScore API**
```
PANDASCORE_TOKEN=<токен доступа к API>
```

**Прочее**
```
AIRFLOW_UID=50000
TZ=UTC
```

> ⚠️ Файл `.env` не включён в репозиторий и не должен публиковаться — он содержит секреты (пароли, токены).

## 🔄 DAG и его задачи

DAG `cybersport_etl_pipeline` запускается ежедневно (`schedule="@daily"`) и включает четыре последовательные задачи:

1. **`extract`** — обращение к PandaScore API за матчами предыдущего дня (`range[begin_at]`) с постраничной загрузкой (пагинация), повторными попытками при сетевых ошибках и HTTP-статусах 429/5xx (реализовано через `tenacity`). Сырые данные сохраняются в MinIO по ключу `raw/{dag_run_id}/matches.json`. Если файл уже существует — повторный запрос к API не выполняется, что обеспечивает идемпотентность при перезапуске задачи.

2. **`transform`** — чтение сырых данных из MinIO, парсинг и валидация с помощью Pydantic-моделей, выделение независимых сущностей: `videogames`, `leagues`, `series`, `tournaments`, `teams`, `matches`, `match_opponents`. Результат сохраняется в MinIO по ключу `transformed/{dag_run_id}/entities.json`.

3. **`load`** — батчевая вставка сущностей в ClickHouse в определённом порядке (сначала справочники, затем зависимые таблицы), с записью статистики в служебные таблицы `etl_run_log` и `etl_state`.

4. **`cleanup`** — удаление временных файлов из MinIO после успешной загрузки.

Каждая задача имеет ограничение по времени выполнения (30 минут) и до 3 автоматических повторов при сбое (кроме `cleanup`, где повтор один).

## 🧪 Тестирование

Для запуска тестов используется отдельный `docker-compose.test.yml`, поднимающий изолированный тестовый экземпляр ClickHouse и контейнер с тестами:

```bash
docker compose -f docker-compose.test.yml up --build --abort-on-container-exit
```

Тесты используют собственный тестовый образ (таргет `test` в `Dockerfile.airflow`) и переменную `PANDASCORE_TOKEN` с заглушкой (`dummy_token_for_tests`), не обращаясь к реальному API. После завершения рекомендуется очистить тестовые контейнеры:

```bash
docker compose -f docker-compose.test.yml down -v
```

## 📈 Мониторинг и алертинг

- **Логирование.** Все модули (`extract`, `transform`, `load`, DAG) используют единую конфигурацию логирования (`common/logging_config.py`), логи пишутся в структурированном виде и доступны через веб-интерфейс Airflow, а также сохраняются в volume `./logs`.
- **`etl_run_log`** — служебная таблица в ClickHouse, фиксирующая каждый запуск задачи `load`: идентификатор запуска, статус (`success`/`failed`), количество извлечённых и загруженных строк, время начала и завершения, текст ошибки при сбое.
- **`etl_state`** — таблица, хранящая последний загруженный идентификатор (`last_id`) и время последней загрузки для каждой сущности, что упрощает отслеживание актуальности данных.
- **Здоровье сервисов.** Все ключевые контейнеры (ClickHouse, PostgreSQL, MinIO, Redis, Superset, все компоненты Airflow) снабжены `healthcheck`, что позволяет docker compose и внешним системам мониторинга отслеживать их состояние.

Отдельной системы алертинга (например, интеграции с Telegram/Slack) в текущей версии проекта не реализовано — при необходимости её можно добавить через `on_failure_callback` в Airflow DAG.

## 🩹 Возможные проблемы и их решение

**Контейнеры Airflow падают с ошибкой прав доступа к volume**
Убедитесь, что переменная `AIRFLOW_UID` в `.env` соответствует UID вашего пользователя (`id -u`), либо используйте значение по умолчанию `50000` и выдайте права на папки `./logs`, `./dags` командой `chmod -R 777` (для локальной разработки).

**ClickHouse не проходит healthcheck при первом запуске**
Инициализация ClickHouse может занимать больше времени на слабом железе — увеличьте `start_period` в healthcheck или дождитесь завершения выполнения `init.sql`.

**Airflow-сервисы зависают в состоянии `unhealthy`**
Проверьте, что `airflow-init` (миграция базы) завершился успешно (`service_completed_successfully`) — остальные сервисы стартуют только после этого.

**Недостаточно памяти при одновременном запуске всех сервисов**
Полный стек (Airflow + ClickHouse + Superset + MinIO + Redis + PostgreSQL) требователен к ресурсам — рекомендуется выделять Docker Desktop не менее 6–8 ГБ RAM.

**Ошибка 429 от PandaScore API**
Учитывайте лимиты API — при большом объёме исторических данных используйте `catchup=False` и запускайте бэкфилл постепенно, либо снизьте частоту параллельных запусков через `max_active_runs`.

**Superset не может подключиться к ClickHouse**
Проверьте, что в `SUPERSET__SQLALCHEMY_DATABASE_URI` указана верная строка подключения и что установлен соответствующий SQLAlchemy-драйвер для ClickHouse.
